# /understand-query

Query codebase knowledge, business domains, wiki, and knowledge graph data via a lightweight CLI tool. Provides a four-layer progressive drill-down model that mirrors the data hierarchy: Business → Interaction → Wiki → KG.

## Options

- `$ARGUMENTS` must contain one subcommand followed by its flags:
  - **Global flags** (before subcommand):
    - `--server URL` — API server base URL (default: `$UNDERSTAND_SERVER` or `http://localhost:3001`)
    - `--token TOKEN` — Access token (required; or set `$UNDERSTAND_TOKEN` env var)
    - `--format json|md` — Output format (default: `json`)
    - `--verbose` — Include additional data (e.g., edges in KG responses)
  - **Subcommands:**
    - `business` — Query business-landscape domains, interactions, and rules
    - `wiki` — Query wiki pages, flows, endpoints, and structure
    - `domain` — Query domain graph nodes and relationships
    - `kg` — Query knowledge graph nodes at source level

---

## Prerequisites

1. **API Server must be running.** This CLI queries the shared API server (same backend as the Dashboard):

```bash
cd understand-anything-plugin/packages/dashboard && pnpm run serve
```

The server prints startup info including an access URL with embedded token:
```
🚀 API Server running at http://localhost:3001
   Access URL: http://localhost:3001?token=<generated-token>
```

2. **Copy the token** and either pass it via `--token` or set the environment variable:

```bash
export UNDERSTAND_TOKEN=<copied-token>
export UNDERSTAND_SERVER=http://localhost:3001   # optional, this is the default
```

3. **Data must be generated.** The API serves data from `.understand-anything/` directories. Ensure relevant skills have been run:
   - `/understand` → generates knowledge graph
   - `/understand-domain` → generates domain graph
   - `/understand-wiki` → generates wiki
   - `/understand-business` → generates business-landscape

## Four-Layer Drill-Down Model

The recommended exploration pattern starts broad (Layer 1) and narrows:

| Layer | Subcommand | Answers |
|-------|-----------|---------|
| 1. Business Overview | `business --list` | What business domains exist across the project? |
| 2. Domain Interactions | `business --domain X --type interactions` | How do users interact with domain X? What steps span server/client? |
| 3. Wiki Detail | `wiki --service S --domain D` | Technical implementation details of domain D in service S? |
| 4. Source-Level KG | `kg --service S --node N` | Class/function structure, relationships, and code location? |

**Example drill-down session:**

```bash
# Layer 1: Discover business domains
python ua_query.py business --list

# Layer 2: Drill into "order" domain interactions
python ua_query.py business --domain order --type interactions

# Layer 3: Check server-side wiki for the order service
python ua_query.py wiki --service order-service --domain order

# Layer 4: Inspect a specific controller class
python ua_query.py kg --service order-service --node OrderController --verbose
```

## Subcommand Reference

### `business` — Business Landscape Queries

Query the cross-facet business-landscape data generated by `/understand-business`.

| Flag | Type | Description |
|------|------|-------------|
| `--list` | boolean | List all business domains with summaries and facet coverage |
| `--domain SLUG` | string | Get full domain detail (interactions, rules, facets) |
| `--type TYPE` | string | Filter domain detail: `interactions` or `rules` |
| `--facet NAME` | string | Get specific facet data for a domain (requires `--domain`) |
| `--search QUERY` | string | Full-text search across all business domain names and summaries |

**Examples:**

```bash
# List all domains (overview)
python ua_query.py business --list

# Get full detail for "order" domain
python ua_query.py business --domain order

# Only interactions for "order"
python ua_query.py business --domain order --type interactions

# Only business rules for "payment"
python ua_query.py business --domain payment --type rules

# Search for anything related to "checkout"
python ua_query.py business --search checkout

# Markdown formatted domain list
python ua_query.py --format md business --list
```

---

### `wiki` — Wiki Data Queries

Query the wiki pages generated by `/understand-wiki`.

| Flag | Type | Description |
|------|------|-------------|
| `--service NAME` | string | Target service name (required) |
| `--type TYPE` | string | Wiki section: `domain`, `endpoint`, `structure`, `flow` |
| `--domain NAME` | string | Specific domain page within a service |
| `--search QUERY` | string | Full-text search across all wiki content |

**Examples:**

```bash
# Get wiki index for order-service
python ua_query.py wiki --service order-service

# Get endpoint documentation
python ua_query.py wiki --service order-service --type endpoint

# Get specific domain page
python ua_query.py wiki --service order-service --domain order

# Search across wiki
python ua_query.py wiki --service order-service --search "payment callback"
```

---

### `domain` — Domain Graph Queries

Query the domain graph generated by `/understand-domain`.

| Flag | Type | Description |
|------|------|-------------|
| `--service NAME` | string | Target service name (required) |
| `--domain NAME` | string | Filter nodes by domain name or ID substring |
| `--search QUERY` | string | Search domain names and summaries |

**Examples:**

```bash
# Get full domain graph for a service
python ua_query.py domain --service order-service

# Find a specific domain
python ua_query.py domain --service order-service --domain order

# Search domains
python ua_query.py domain --service order-service --search "user"
```

---

### `kg` — Knowledge Graph Queries

Query the knowledge graph generated by `/understand`.

| Flag | Type | Description |
|------|------|-------------|
| `--service NAME` | string | Target service name (required) |
| `--type TYPE` | string | Filter nodes by type: `class`, `interface`, `function`, `module`, `file`, etc. |
| `--node NAME` | string | Find a specific node by exact name |
| `--search QUERY` | string | Full-text search across all node data (names, labels, metadata) |
| `--file PATH` | string | Read source file with graph annotations (server-side file read) |

**Examples:**

```bash
# List all class nodes
python ua_query.py kg --service order-service --type class

# Find specific class
python ua_query.py kg --service order-service --node OrderController

# Full-text search
python ua_query.py kg --service order-service --search "validate"

# Include edges in output
python ua_query.py --verbose kg --service order-service --node OrderController

# Read source file with annotations
python ua_query.py kg --service order-service --file src/controllers/OrderController.java
```

## Error Handling

| Exit Code | Meaning | Recommended Action |
|-----------|---------|-------------------|
| 0 | Success | Output printed to stdout |
| 1 | Client error | Check arguments, API returned 4xx |
| 2 | Server unavailable | Start API server: `pnpm run serve` |

**Error messages are printed to stderr.** Successful JSON/markdown output goes to stdout.

Common error scenarios:

| Scenario | Behavior |
|----------|----------|
| `--token` not provided and `$UNDERSTAND_TOKEN` unset | Exit 1 with usage hint |
| Server not running (connection refused) | Exit 2 with startup instructions |
| Subcommand requires `--service` but not provided | Exit 1 with `SystemExit` message |
| API returns 404 (data not generated yet) | Exit 1, print HTTP error |
| API returns 500 (server error) | Exit 1, print error body |

## Output Formats

### JSON (default)

Standard JSON output with `indent=2`, suitable for piping to `jq` or consumption by other tools:

```bash
python ua_query.py business --list | jq '.domains[].name'
```

### Markdown (`--format md`)

Human-readable markdown, useful for embedding in agent responses:

- Domain lists → `## Domain Name` with summary
- Search results → bullet list with bold names
- Other data → fenced JSON code block

## Integration with Agent Workflow

This skill is designed for AI agents to programmatically query knowledge. Typical agent patterns:

1. **Contextual lookup:** Agent needs to understand a service before making changes
2. **Cross-reference:** Agent checks business rules before modifying domain logic
3. **Search:** Agent searches for related components across the knowledge base
4. **Validation:** Agent verifies domain coverage after running `/understand-business`

**Chaining example (agent script):**

```bash
# 1. Check if business-landscape exists
python ua_query.py business --list 2>/dev/null
if [ $? -eq 2 ]; then
  echo "Server not running — starting..."
  cd understand-anything-plugin/packages/dashboard && pnpm run serve &
  sleep 3
fi

# 2. Find relevant domain
DOMAIN=$(python ua_query.py business --search "payment" | jq -r '.results[0].id // empty')

# 3. Get interactions
python ua_query.py business --domain "${DOMAIN#domain:}" --type interactions
```
